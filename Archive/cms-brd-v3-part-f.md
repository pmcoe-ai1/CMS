### Part F: Non-Functional Requirements

### 5.21 Performance & Speed

| ID | Requirement | Traditional Interface | AI Interface |
|----|-------------|----------------------|-------------|
| 5.21.1 | All pages must load in under 3 seconds for users in the website's designated target regions. | The performance dashboard shows page load times per page with pass/fail indicators against the 3-second threshold. Pages exceeding the threshold are highlighted with diagnostic details. | User says "which pages are loading slower than 3 seconds?" and the system lists them with load times. User says "what's the average page load time across the site?" and the system reports. |
| 5.21.2 | The system must meet or exceed Google Core Web Vitals thresholds (Largest Contentful Paint, Cumulative Layout Shift, Interaction to Next Paint) on all pages. Core Web Vitals affect both Google AI Overviews and crawler efficiency. | The Core Web Vitals panel (per 5.14.6) shows LCP, INP, and CLS scores per page with field data from Google Search Console and lab data from built-in testing. | User says "show me Core Web Vitals for our top 10 pages" and the system displays scores. User says "what's causing the high CLS on the homepage?" and the system diagnoses contributing elements. |
| 5.21.3 | The system must support content delivery from geographically distributed servers or CDN nodes to minimise latency for users across all target regions. | User opens the CDN configuration panel in deployment settings. Views CDN node locations, cache status, and latency metrics per region. | User says "show me CDN latency by region" and the system displays metrics. User says "purge the CDN cache for the pricing page" and the system invalidates it. |
| 5.21.4 | The system must support caching strategies that serve pages quickly without sacrificing content freshness. | User opens the caching settings panel. Configures cache duration per content type, cache invalidation rules, and views cache hit rates. | User says "what's our cache hit rate?" and the system reports. User says "set blog post cache to 1 hour and product pages to 5 minutes" and the system applies the rules. |
| 5.21.5 | The user must be able to designate target geographic regions for each website, and the system must optimise delivery accordingly. | User opens the regional targeting panel in site settings. Selects target regions from a map or list. The system configures CDN routing and performance monitoring for selected regions. | User says "set target regions to Australia, New Zealand, and Singapore" and the system configures delivery optimisation for those regions. User says "what are our current target regions?" and the system lists them. |

### 5.22 Deployment & Platform

| ID | Requirement | Traditional Interface | AI Interface |
|----|-------------|----------------------|-------------|
| 5.22.1 | The user must be able to specify their preferred hosting platform for deployment. The system must not mandate a specific hosting provider. | User opens the deployment settings panel. Selects from supported hosting platforms (AWS, Google Cloud, Azure, managed hosting, self-hosted) or enters custom deployment configuration. | User says "deploy this site to AWS" and the system opens the AWS deployment configuration. User says "what hosting platform are we currently using?" and the system reports. |
| 5.22.2 | The system must support custom domain names with SSL/HTTPS certificate configuration. | User opens the domain settings panel. Adds custom domain names. Configures or auto-provisions SSL certificates (Let's Encrypt or custom). DNS configuration instructions are displayed. | User says "add the domain www.acme.com to this site" and the system opens domain configuration with DNS instructions. User says "is our SSL certificate valid?" and the system checks and reports status and expiry. |
| 5.22.3 | The system must support CDN integration for content delivery optimisation across geographically dispersed target markets. | User opens the CDN integration panel. Selects a CDN provider (Cloudflare, AWS CloudFront, Fastly, or equivalent) or configures a custom CDN. Views integration status and performance metrics. | User says "connect Cloudflare as our CDN" and the system opens the integration setup. User says "is the CDN active for all regions?" and the system reports coverage. |
| 5.22.4 | The system must be hosting-platform-agnostic — deployable to major cloud providers (AWS, Google Cloud, Azure), managed hosting platforms, or self-hosted infrastructure without vendor lock-in. | The deployment panel shows platform-agnostic configuration options. Migration tools allow moving between hosting platforms without content loss. | User says "can we move from AWS to Google Cloud?" and the system confirms compatibility and describes the migration process. User says "generate a deployment package for self-hosted infrastructure" and the system creates it. |

### 5.23 Upgradability

| ID | Requirement | Traditional Interface | AI Interface |
|----|-------------|----------------------|-------------|
| 5.23.1 | The system must support updates and upgrades without disrupting the live website. | The update manager panel shows available updates with type (security, feature, AI model), risk level, and release notes. User clicks "Apply Update" and the system performs a zero-downtime update. | User says "are there any updates available?" and the system lists them with details. User says "apply the latest security patch" and the system performs the update without downtime. |
| 5.23.2 | Core system updates (security patches, feature additions, AI model improvements) must be applicable without requiring the user to rebuild content or reconfigure settings. | After an update, the system displays a confirmation showing what was updated and verifying that all content and settings remain intact. A post-update health check runs automatically. | User says "did the update affect any of our pages or settings?" and the system runs a post-update verification and reports. |
| 5.23.3 | The system must maintain backward compatibility so that existing pages and configurations continue to work after upgrades. | The update manager shows a compatibility check before each update, listing any potential breaking changes. User reviews and approves before proceeding. | User says "will this update break anything on our site?" and the system runs a compatibility check and reports any risks. |
| 5.23.4 | The system must provide a staging or preview environment where upgrades can be tested before being applied to the live site. | User opens the staging environment panel. Creates a staging copy of the live site. Applies updates to staging first. Compares staging and live side by side. Promotes staging to live when satisfied. | User says "create a staging copy of the site" and the system provisions it. User says "apply the update to staging only" and the system applies it. User says "compare staging and live" and the system shows differences. User says "promote staging to live" and the system deploys. |
| 5.23.5 | The system must support versioning of its own components so that users can understand what version they are running and what improvements are available. | The system information panel shows current versions of all components (core CMS, AI models, plugins, theme engine) with available update indicators. A changelog is accessible for each component. | User says "what version of the CMS are we running?" and the system reports. User says "what's new in the latest version?" and the system displays the changelog. |

### 5.24 Portability

| ID | Requirement | Traditional Interface | AI Interface |
|----|-------------|----------------------|-------------|
| 5.24.1 | All website content (pages, blog posts, images, files, forms, settings) must be exportable in standard, open formats at any time. | User opens the data export panel. Selects content categories to export (all, pages only, blog only, media only). Chooses an export format. Initiates the export and downloads the archive. | User says "export all our website content" and the system generates a downloadable archive. User says "export just the blog posts" and the system generates a filtered export. |
| 5.24.2 | The system must support export to common CMS formats including WordPress (WXR/XML), static HTML, and structured JSON/Markdown. | The export format selector offers WordPress WXR/XML, static HTML, JSON, and Markdown options. Format descriptions explain what each includes and what it's compatible with. | User says "export the site as WordPress XML" and the system generates the WXR file. User says "export as static HTML" and the system generates the HTML archive. |
| 5.24.3 | Exported content must include all associated metadata (meta titles, descriptions, schema markup, URL structures, image alt text, publish dates, author information). | The export preview panel shows a summary of included metadata per content item. A completeness check verifies that all metadata is present in the export before download. | User says "does the export include all SEO metadata?" and the system verifies and reports. User says "export with full metadata including schema markup" and the system includes everything. |
| 5.24.4 | The system must not impose proprietary formats that prevent content from being used in other systems. | All export formats are documented open standards. The export panel displays format specifications and compatibility notes. | User says "can I import this export into WordPress?" and the system confirms compatibility for the WXR format. |
| 5.24.5 | The user must retain full ownership of all content created within the system. No content is locked to the platform. | The data ownership policy is displayed in the site settings panel. Export functionality is always available regardless of account status or subscription tier. | User says "do we own all our content?" and the system confirms full ownership with no platform lock-in. |
| 5.24.6 | Database and file storage must be exportable independently for backup or migration purposes. | User opens the backup/export panel. Exports the database separately from the file storage (media, assets). Both are available as independent downloadable archives. | User says "export just the database" and the system generates the database export. User says "export the media library separately" and the system generates the file archive. |
| 5.24.7 | The system must support domain transfer without content loss if the user moves to a different deployment arrangement. | User opens the domain transfer panel. Follows guided steps for DNS migration. The system verifies content integrity after the transfer. | User says "we're moving to a new hosting provider — what steps are needed for domain transfer?" and the system provides the migration guide with verification steps. User says "verify that no content was lost after the migration" and the system runs an integrity check. |
